You don't have to wire a paramblock parameter to 
an interface item - 
wiring parameter blocks to UI rollouts is 
optional.
You can simply read/write the parameter block's value as
if it 
was a local variable for the plug-in. (It is actually
a kind of local 
variable with the difference that it is being
managed automatically, can 
appear in trackview and is stored 
with the plug-in in the scene.

Dave,

looks like a very cool 
script!
to solve the problem, i would create a simple function to update the 
buttons states, istead of using the oldBtn parameter (so that you won't depend 
on the plugin history). something like:


fn updateBnStates n = (

    chkbn1.state = n==1

    chkbn2.state = n==2

    .

    .

)




now, you don't need the "on matID set val" that you have now. when 
you use the ui parameter in the parameters block, it hard-wires the ui and the 
parameter. you do need it to set the buttons states.



on MatID set val do params.updateBnStates val

Dave,

the first part of my answer was 
as you described, just another way of doing what you have done before.
the 
second part refers to the problem at hand.
i'll try to explain it again, 
since i wasn't clear in my first explenation.

in your code you had this 
line in the parameter block:
on matID set val do delegate.materialID = val 


this line is not needed, since when you use the ui parameter 
in the declerations in the parameters block, MXS automatically hard-wires that 
parameter and the appropriate ui element.

now, you CAN use the 
"on matID set val ..." to change the other parts of the ui. the "matID 
set" handler will be called whenever the value of matID is changed, whether 
direcly (using ui or assigning a value in MXS) or indirectly when moving the 
time slider and the value is animated. so putting this line in the parameter 
block should solve you problem:
on matID set val do params.updateBnStates 
val

o
